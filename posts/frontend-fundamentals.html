공용 컴포넌트 설계 방법 고찰 (Frontend Fundamentals 정리)

2025.01.29 · minnote29

공용 컴포넌트를 설계하기 위해 Frontend Fundamentals 내용을 참고했다. 읽어보면서 프론트엔드 코드 구조를 고민할 때 유용하다고 느껴 따로 정리해보았다.

변경하기 쉬운 코드

좋은 프론트엔드 코드는 결국 변경하기 쉬운 코드라고 할 수 있다. 서비스는 계속 변화하기 때문에 새로운 요구사항이 추가되거나 기존 로직을 수정해야 하는 상황이 계속 발생한다.

Frontend Fundamentals에서는 코드 변경을 쉽게 하기 위한 기준으로 다음 네 가지 요소를 제시한다.

  • 가독성
  • 예측 가능성
  • 응집도
  • 결합도

1. 가독성

가독성은 코드가 얼마나 읽기 쉬운지를 의미한다. 코드를 변경하기 위해서는 먼저 코드가 어떤 동작을 하는지 이해할 수 있어야 한다.

가독성을 높이기 위한 전략은 다음과 같다.

  • 중복 코드 지양하기
  • 맥락 줄이기
  • 복잡한 조건에 이름 붙이기
  • 매직 넘버 제거하기
  • Top-down 방식으로 읽히는 코드 구조 만들기

특히 매직 넘버는 코드의 의미를 파악하기 어렵게 만들기 때문에 상수로 선언해 의미를 명확히 하는 것이 좋다.

2. 예측 가능성

예측 가능성은 다른 개발자가 함수나 컴포넌트의 동작을 얼마나 쉽게 예상할 수 있는지를 의미한다.

예측 가능성을 높이기 위한 전략은 다음과 같다.

  • 이름이 겹치지 않도록 관리하기
  • 같은 종류의 함수는 반환 타입을 통일하기
  • 숨은 로직을 드러내기

예를 들어 API 호출 관련 훅들이 서로 다른 반환 타입을 가지면 사용하는 입장에서 혼란이 발생할 수 있다.

3. 응집도

응집도는 함께 수정되어야 할 코드가 실제로 함께 수정되는지를 의미한다.

특정 로직이 여러 컴포넌트에서 사용되더라도 함께 수정되어야 하는 코드라면 하나의 구조 안에서 관리하는 것이 좋다.

응집도를 높이기 위한 전략은 다음과 같다.

  • 함께 수정되는 파일을 같은 디렉토리에 배치하기
  • 매직 넘버 제거하기
  • 폼 단위 응집도 고려하기

폼의 경우 다음 두 가지 방식으로 응집도를 관리할 수 있다.

  • 필드 단위 응집도: 각 입력 요소가 독립적으로 관리되는 방식
  • 폼 전체 단위 응집도: 모든 검증 로직이 폼에 종속되는 방식

4. 결합도

결합도는 코드를 수정했을 때 영향을 받는 범위를 의미한다. 영향 범위가 작을수록 코드의 예측 가능성이 높아진다.

결합도를 낮추기 위한 전략은 다음과 같다.

  • 책임을 하나씩 관리하기
  • 불필요한 공통화 피하기
  • Props Drilling 줄이기

여러 컴포넌트에서 사용하는 로직을 무조건 공통화하는 것이 항상 좋은 선택은 아니다. 공통화된 코드가 많아질수록 변경 시 영향 범위가 넓어질 수 있기 때문이다.

또한 Props Drilling은 컴포넌트 간 결합도가 높다는 신호이므로 Composition 패턴 등을 활용해 구조를 개선할 수 있다.

정리

Frontend Fundamentals에서 말하는 좋은 프론트엔드 코드는 결국 다음 네 가지 기준의 균형을 맞추는 것이다.

  • 가독성
  • 예측 가능성
  • 응집도
  • 결합도

이 네 가지를 고려하며 컴포넌트와 구조를 설계한다면 장기적으로 유지보수하기 쉬운 프론트엔드 코드를 만들 수 있다.
